# Nexus

github copilot可使用的多 Agent 编排系统。核心目标：**降上下文成本、降幻觉、降遗忘，让强模型做判断，让快速模型做执行。**

---

## github copilot 第三方模型使用
可通过插件 OAI Compatible Provider for Copilot 配置第三方模型

---

## 项目要解决的问题

单模型跑长开发任务时，有几个典型瓶颈：

1. **上下文越来越重**：仓库、对话历史、日志全灌进去，越往后越不稳定，成本也高。
2. **一边研究一边实现一边自查**：一个模型什么都干，容易自圆其说、漏掉边界和调用方。
3. **中途遗忘**：链长了就忘了前面的结论，多轮修改容易偏离最初目标。
4. **所有环节都用强模型，成本不划算**：研究清楚后的实现、迁移、校验其实确定性很高，没必要上最贵的模型。

---

## 核心思路

把长任务拆成多个短上下文子任务，用文件化的 artifact 沉淀推理结果，后续 Agent 按结果执行，而不是反复阅读所有历史和代码。

核心原则只有几个：
- **研究与实现分开**：先研究清楚，再让实现 Agent 照做。
- **UI 与逻辑分开**：呈现层和业务逻辑由不同 Agent 负责。
- **文件化交接**：结论写进报告，不靠聊天记忆传递。
- **主编排器只保留最小上下文**：Nexus 不看大段代码和分析，只读摘要和 artifact 路径。
- **合适模型做合适的事**：强模型做研究、判断，性价比模型做确定性执行和验证。

## Agent 分工

| Agent | 职责 | 说明 |
| --- | --- | --- |
| Nexus | 主控 | 分诊、路由、阶段推进、恢复，不吃重内容 |
| Investigator | 逻辑研究 | 契约发现、影响半径、Implementation-Ready 报告 |
| UI_Investigator | UI 研究 | 布局、状态、响应式、无障碍蓝图 |
| Coder | 逻辑实现 | 按研究报告实现后端/业务逻辑 |
| UI_Coder | UI 实现 | 只做呈现层，不写业务逻辑 |
| Reviewer | 独立 QA | 静态审查 + 自动化验证，不做审美判断 |
| WebSearcher | 外部检索 | 唯一可做网络搜索的 Agent，带缓存 |
| DocWriter | 文档 | 按需更新 README、doc、CHANGELOG |

## 任务分级

先判级再决定流程深度，避免所有任务走一套重流程。

| 等级 | 场景 | 默认流程 |
| --- | --- | --- |
| T0 | 极小确定性改动 | Fast Lane，快速执行 |
| T1/T2 | 范围明确/跨模块但不模糊 | 直接进入实现级研究 → 实现 → Review |
| T3/T4 | 需求不清或高风险 | 先初步研究 → 用户确认 → 实现级研究 → 实现 → 深度 Review |

---

## 标准工作流

```text
用户任务
  → Nexus 分诊
  → Investigator / UI_Investigator 研究（输出 Implementation-Ready 报告）
  → Coder / UI_Coder 实现
  → Reviewer 验证
  → DocWriter 按需更新文档
  → 完成
```

T3/T4 任务会在研究前多加一次 Preliminary 和用户确认环节。

## 目录结构

```text
.Nexus/
  plan.md                    # 当前任务、阶段、恢复点
  0-research/                # 研究报告
  1-reviewer/                # QA 报告 / 手动验证清单
  2-implementation/          # 实现报告
  .search-cache/             # Web 搜索缓存
  .tool/                     # 辅助脚本
```

所有关键产物都落盘，可追溯、可恢复。

## 为什么能降低上下文占用

- 主编排器不读写长内容，只碰摘要和路径。
- 研究结论落盘成报告，不需要反复"全文阅读"。
- 实现 Agent 只读实现级报告，不自己做一遍研究。
- 大文件先走辅助脚本做结构化提取，不直接灌进模型。
- 外部搜索又缓存又统一，减少重复信息摄入。

整体效果：长任务被切成多个短上下文任务，而不是一个模型拖着所有历史跑全程。

---

## 为什么能降低幻觉和遗忘

- **角色隔离**：不同 Agent 只管自己这一段，减少"边猜边做"。
- **未知项显式阻塞**：契约不清、语义不明就停下来，不求强写结论。
- **报告化交接**：关键结论有文件记录，不会被对话冲掉。
- **独立 Reviewer**：实现方不自证，Review 用对抗视角检查。
- **外部信息统一入口**：减少不同 Agent 乱引用低质来源。

## 成本策略

- 强模型：负责研究、判断、影响分析等高不确定任务。
- 性价比模型：负责已经研究清楚的实现、迁移、验证、文档等确定性工作。

前提是研究报告必须足够清晰，否则不会硬塞给便宜模型。

### 缓存命中与多 Agent 的取舍

多 Agent 会影响底层模型的缓存命中。不同 Agent 的 system prompt 不同，调用也被拆成多次独立请求，所以 **prompt cache / prefix cache** 不一定比单 Agent 更好。

本项目的优势不主要靠模型缓存，而是靠 **减少无效上下文** 和 **复用中间产物**：

- 研究、实现、审查分阶段进行，每个 Agent 只读取当前阶段需要的信息
- 关键结论会写成报告，而不是反复塞进对话上下文
- 外部搜索有缓存，避免重复检索
- 任务打回时通常只需要局部重跑，不需要整条链路重来

- **短任务**：单 Agent 往往更直接，缓存也可能更友好
- **长任务 / 大仓库 / 多阶段任务**：多 Agent 通常更有优势，因为它能降低总输入、减少重复推理，并且更容易恢复和复用

## 接入前提

这不是即开即用的独立产品，是一个多 Agent 工作流和角色规范。

需要你具备：
- 支持多 Agent 的运行环境
- 文件读写、搜索、终端、Web 检索等工具
- 可用的底层模型
- 按角色分配工具和权限的基础设施

推荐接入步骤：导入 Agent 定义 → 把 Nexus 设为入口 → 用一个小范围 T1/T2 任务跑通闭环 → 逐步扩大适用范围。

---

## 适合/不适合

**适合**：多阶段开发大仓库、调用链复杂的重构、UI/逻辑需要拆分的任务、易中断的长任务。

**不适合**：几行改动的小任务、无文件系统的对话环境、只求单轮出答案的用法。

---

## 当前限制

- 依赖底层模型遵守角色约束
- 依赖运行环境支持多 Agent、文件、工具
- 研究报告质量直接影响下游结果
- 超小任务可能反而增加流程开销
- 专为开发任务设计，不是通用聊天框架
